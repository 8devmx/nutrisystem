# Refactorización de módulo de planes

## Contexto
Necesitamos refactorizar el módulo de planes para que el plan que se construya sea para toda la duración del plan, es decir, no se deberá de generar un plan por día, sino un plan para la semana, quincena, mes.

## Tareas (Checklist)
- [x] Eliminar en la ventana de nuevo plan, la intefaz para cargar los días del 2 en adelante, es decir, solo se tiene que rellenar el primer día.
- [x] Insertar plan y editar plan deberán tener la misma lógica, usabilidad y experiencia de usuario.
- [x] Agregar pills de sugerencias en cada sección de las comidas, por ejemplo: pills de recetas que tengan la categoría de desayunos, cerca de desayuno

## Bugs corregidos (18/03/2026 — segunda iteración)

### Bug 1 — Error de servidor al cargar usuarios
**Causa:** `fetchData` usaba `Promise.all` para cargar usuarios, alimentos y unidades. Si cualquiera de las tres peticiones fallaba, la promesa entera rechazaba y el bloque `catch` capturaba el error sin ejecutar ningún `setState`, dejando el formulario sin datos.

**Fix (`new/page.js`):** Se cambió `Promise.all` por `Promise.allSettled`. Ahora cada petición se evalúa de forma independiente. Si usuarios falla, se muestra un toast de error y el array queda vacío, pero alimentos y unidades se cargan correctamente. El formulario siempre termina de inicializar.

### Bug 2 — Usuarios no aparecen en el selector
**Causa:** Consecuencia directa del bug 1. Al rechazar el `Promise.all`, `setUsers` nunca se ejecutaba.

**Fix:** Resuelto con el mismo cambio a `Promise.allSettled`.

### Bug 3 — Seguían apareciendo todos los días en lugar de solo el Día 1
**Causa:** El borrador guardado en `sessionStorage` con la versión anterior del código contenía claves para todos los días (`2-breakfast`, `3-lunch`, etc.). Al restaurar el draft, esas claves se cargaban en el estado `meals` y el template las renderizaba aunque el nuevo código solo muestra el Día 1 en pantalla — pero las entradas `2-*`, `3-*`, etc. seguían activas en memoria y se guardaban al hacer submit.

**Fix (`new/page.js`):** Al restaurar el borrador, se filtra `savedMeals` para conservar únicamente las entradas cuya clave empieza con `'1-'`. Los días 2+ del borrador antiguo se descartan silenciosamente.

### Bug 4 — Pills de recetas no aparecían
**Causa:** El filtro buscaba `r.category` (string), campo que no existe en el modelo `Recipe`. El campo real es `meal_categories` (array), definido en el modelo Eloquent con `$casts = ['meal_categories' => 'array']` y los valores del enum `MealCategory`: `breakfast`, `snack`, `lunch`, `dinner`.

**Fix (`new/page.js` y `edit/page.js`):** Se reemplazó el filtro por string por una comprobación de array, usando un `categoryMap` que traduce los momentos del plan (`morning_snack`, `afternoon_snack`) al valor `snack` del enum:
```js
const categoryMap = {
  breakfast:       'breakfast',
  morning_snack:   'snack',
  lunch:           'lunch',
  afternoon_snack: 'snack',
  dinner:          'dinner',
}
Array.isArray(r.meal_categories) && r.meal_categories.includes(targetCategory)
```

## Cambios realizados

### Archivos modificados
- `frontend/src/app/admin/plans/new/page.js`
- `frontend/src/app/admin/plans/[id]/edit/page.js`

---

### Tarea 1 — Solo Día 1 en Nuevo Plan (`new/page.js`)

**Problema:** El formulario renderizaba todos los días del plan (7, 14 o 30), haciendo la UI muy larga e innecesaria, ya que todos los días eran idénticos.

**Cambios:**
- Se eliminó el `Array.from({ length: totalDays })` que iteraba y renderizaba todos los días.
- Ahora solo se muestra el **Día 1** con una etiqueta visual "Plantilla del plan".
- Se agregó un banner informativo que explica al usuario que el Día 1 se replicará automáticamente para todos los días del plan al guardar.
- Se refactorizó `handleSubmit`: en lugar de leer comidas de todos los días desde el estado, ahora toma únicamente las entradas con clave `1-*` y genera las comidas para cada día del 1 al `totalDays` en el momento del guardado.

---

### Tarea 2 — Unificación de UX Insertar/Editar (`edit/page.js`)

**Problema:** La página de edición trataba los alimentos ya guardados (`_existing: true`) como de solo lectura, a diferencia de la página de creación donde todo es editable. Esto generaba una experiencia inconsistente.

**Cambios:**
- Se eliminaron los `readOnly` y los guards `if (item._existing) return` de los inputs de alimento y cantidad.
- Los alimentos existentes ahora son completamente editables: se puede cambiar el alimento buscando uno nuevo, modificar la cantidad y cambiar la unidad.
- Al seleccionar un nuevo alimento sobre uno existente, el item se marca como `_existing: false` para que el flujo de guardado lo trate como nuevo.
- El select de unidad ahora resuelve también `item.unit?.id` para los ítems existentes que no tenían `unit_id` directo en el estado.
- En `handleSubmit` se agregó el **paso 2** de actualización: recorre todos los ítems `_existing` y hace `PUT /v1/plans/{id}/meals/{meal_id}` para persistir los cambios de cantidad.
- La lógica de guardado de nuevas comidas ahora resuelve la unidad con `item.unit_id || item.unit?.id` para cubrir ambos casos (existente editado vs nuevo).

---

### Tarea 3 — Pills de sugerencias de recetas (ambas páginas)

**Problema:** No había forma rápida de agregar recetas conocidas a una sección de comida sin buscarlas manualmente.

**Cambios (aplicados en `new/page.js` y `edit/page.js`):**
- En cada sección de comida (Desayuno, Colación mañana, Comida, Colación tarde, Cena) se renderizan **pills clicables** con las recetas cuya `category` corresponde a ese momento del día.
- El filtrado mapea palabras clave en español e inglés: `desayuno`, `colaci`, `snack`, `comida`, `almuerzo`, `cena`, `dinner`.
- Máximo **6 pills** por sección para no saturar la UI.
- Al hacer clic en una pill, la receta se agrega directamente a esa sección usando `addRecipeToMeal`.
- Las pills solo se muestran si hay recetas que coincidan con la categoría del momento.
